## 一、什么是蓝牙

**定义**：蓝牙是一种工作在 **2.4 GHz ISM 频段**上的短距离无线通信技术，可实现固定设备与移动设备之间的数据交换。

### 两种蓝牙类型

| 类型 | 说明 | 频段 | 典型应用 |
|------|------|------|----------|
| **传统蓝牙（BR/EDR）** | 蓝牙 3.0 及之前版本，高速率 | 2.4 GHz | 音频传输、大文件传输 |
| **低功耗蓝牙（BLE）** | 蓝牙 4.0 起支持的新架构 | 2.4 GHz | 智能手环、IoT 设备 |

> 💡 蓝牙 4.0 并不是蓝牙 3.0 的简单升级，而是**全新的技术架构**。两者在物理层、链路层协议上完全不兼容。

---

## 二、BLE 定义与特点

**BLE = Bluetooth Low Energy（低功耗蓝牙）**

### 核心特点

| 特点 | 详细说明 |
|------|----------|
| 🟢 **超低功耗** | 使用纽扣电池即可运行**数月至数年**（对比传统蓝牙数天） |
| ⚡ **快速连接** | **毫秒级**的连接速度（传统蓝牙可能需要数秒至数分钟） |
| 📡 **远距离通信** | 最远可达**数百米**（传统蓝牙通常仅 10 米左右） |
| 🔒 **安全性高** | 支持 AES-128 加密 |

## 三、BLE 协议栈架构

BLE 协议栈采用分层设计，自底向上如下：

```
┌─────────────────────────────────────────────┐
│              应用层 (Application)            │
├──────────┬──────────┬───────────────────────┤
│   GATT   │   GAP    │    （上层协议）         │
│ (通用属性)│ (通用访问)│                       │
├──────────┴────┬─────┴───────────────────────┤
│     ATT       │         SMP                  │
│  (属性协议)    │     (安全管理协议)             │
├───────────────┴─────────────────────────────┤
│                L2CAP                          │
│      (逻辑链路控制与适配协议)                   │
├─────────────────────────────────────────────┤
│                 HCI                           │
│           (主机控制器接口)                      │  ← SoC方案中为内部API
├─────────────────────────────────────────────┤
│          LL (链路层 Link Layer)               │
├─────────────────────────────────────────────┤
│          PHY (物理层 Physical Layer)          │
└─────────────────────────────────────────────┘
```

### 各层功能速查

| 层级 | 名称 | 核心职责 |
|:----:|------|---------|
| PHY | 物理层 | 无线电波收发，调制解调，频率选择 |
| LL | 链路层 | 状态机管理，广播/扫描/连接，空中接口包格式 |
| HCI | 主机控制器接口 | 主机(Host) 与 控制器(Controller) 之间的标准化接口（SoC 方案下为内部 API） |
| L2CAP | 逻辑链路控制与适配 | 协议复用(CID)，数据分片重组，流控 |
| ATT | 属性协议 | Client-Server 模型的属性读写框架 |
| GATT | 通用属性配置文件 | 组织属性的层次化结构（Service → Characteristic → Value） |
| GAP | 通用访问配置文件 | 定义设备角色和连接建立过程 |
| SMP | 安全管理协议 | 配对(Pairing)、加密(AES-128 CCM)、密钥管理 |

> **对于应用开发者**：重点掌握 **GAP + GATT + ATT** 三层。
> **对于驱动/协议栈开发者**：需要深入理解 **PHY + LL + L2CAP**。

## 四、物理层 (PHY)

### 4.1 频段与信道

| 参数 | 值 |
|------|---|
| **频段范围** | 2400 MHz ~ 2483.5 MHz（ISM 频段） |
| **信道总数** | **40 个**物理信道 |
| **信道间距** | 2 MHz |
| **调制方式** | GFSK（高斯频移键控），基础速率 1 Mbps；BLE 5.0 新增 2 Mbps 选项 |

### 4.2 两类信道

```
信道分布：
┌──────────────────────────────────────────────────────────────┐
│  0   1   2   ...  36 │ 37      38      39                    │
├─────────────────────────────┼─────────────────────────────────┤
│     数据信道（37个）          │  广播信道（3个）                  │
│  用于连接后通信 + 自适应跳频    │  用于设备发现和广播              │
│                              │  选择避开 Wi-Fi 干扰频段           │
└─────────────────────────────┴─────────────────────────────────┘
```

| 信道类型 | 数量 | 用途 | 编号 |
|----------|------|------|------|
| **数据信道** | 37 个 | 连接后数据传输，自适应跳频(AFH) | 0~36 |
| **广播信道** | 3 个 | 广播/发现设备，避让 WiFi 干扰 | 37, 38, 39 |

> 📌 **为什么选 37/38/39？**
> 这三个信道位于 2.4 GHz 频段的边缘位置，恰好避开了 WiFi（802.11）最常用的 1/6/11 信道区域，减少同频干扰。

## 五、链路层角色

BLE 设备有四种角色：

| 角色 | 英文 | 行为能力 | 典型场景 |
|------|------|----------|----------|
| **主机 / 中央设备** | Master / Central | 发起连接请求 | 手机、PC |
| **从机 / 外围设备** | Slave / Peripheral | 广播并等待被连接 | 智能手环、传感器 |
| **广播者** | Broadcaster | 只能持续广播，不能被连接 | Beacon 信标 |
| **观察者** | Observer | 只监听广播，不能发起连接 | 扫描器 |

> ⚠️ **主从一体**：同一 BLE 设备可以同时作为主机和从机，每个设备都是对等的，更加实用灵活。

### 数据流向示意

```
主机 (Master/Central)                     从机 (Slave/Peripheral)
  ┌──────────────┐                          ┌──────────────┐
  │  发起连接请求  │  ──── CONNECT_REQ ──→   │  广播等待连接  │
  │  主动请求数据  │  ←─── ATT Read/Write ──  │  响应数据     │
  │  接收通知     │  ←─── Notify/Indicate ── │  主动推送数据  │
  └──────────────┘                          └──────────────┘
```

## 六、广播机制

### 6.1 什么是广播？

用夜市比喻理解：
- **小贩（广播者）**：周期性地喊出商品信息（广播包）
- **顾客（观察者）**：用耳朵听各种广告信息（扫描）
- 为了避免所有人在同一个频道喊导致混乱，使用 **37、38、39** 三个固定通道
- 小贩不能一直喊（耗电+干扰），所以有了**广播间隔**

### 6.2 广播事件流程

```
从机 (Slave) 广播过程：

  t=0ms        t=0ms        t=0ms
  ┌─────┐    ┌─────┐    ┌─────┐
  │CH37 │    │CH38 │    │CH39 │
  │发送 │ →  │发送 │ →  │发送 │
  └─────┘    └─────┘    └─────┘
       ↑ 一个广播事件（3个信道依次发送）
       │
       ▼
  ┌─────────────┐
  │  随机延时     │  0 ~ 10 ms（避免碰撞）
  └─────────────┘
       │
       ▼
  ┌─────────────┐
  │  等待下一个   │  广播间隔 (20ms ~ 10.24s)
  │  广播事件...  │
  └─────────────┘
```

### 6.3 关键参数

| 参数 | 取值范围 | 说明 |
|------|----------|------|
| **广播间隔** | 20 ms ~ 10.24 s | 单位：0.625 ms（即 32 ~ 16384 个 slot） |
| **随机延迟** | 0 ~ 10 ms | 两个广播事件之间的随机延时，防碰撞 |
| **实际间隔** | interval ~ interval+10ms | 设置 100ms 时实际可能是 100~110ms |

> ⚠️ **广播间隔越大 → 功耗越低，但建立连接的时间越长**（需要在正确时机捕获到广播包）

### 6.4 广播数据包

| 项目 | 蓝牙 4.x | 蓝牙 5.0+ |
|------|----------|-----------|
| **最大载荷** | **31 字节** | **255 字节**（通过扩展广播实现） |

#### 广播数据结构（AD Structure）

每个广播数据由若干个 **AD 结构体** 组成：

```
AD Structure 格式（TLV 编码）：
┌──────────┬──────────┬──────────────────┐
│ Length   │ AD Type  │ Data             │
│ (长度)   │ (类型)   │ (数据)           │
│ 1 byte   │ 1 byte   │ Length - 1 bytes │
└──────────┴──────────┴──────────────────┘
```

#### 常用 AD Type 列表

| AD Type 值 | 名称 | Data 含义 |
|:----------:|------|-----------|
| `0x01` | **Flags（标志位）** | LE 发现模式、BR/EDR 支持等（见下方详细定义） |
| `0x02` | 不完整本地名称 | 设备名缩写 |
| `0x03` | 完整本地名称 | 设备全名 |
| `0x08` | 完整本地名称 | 同 0x03 |
| `0x09` | 缩写本地名称 | 设备名的简称（必须是全名前缀） |
| `0xFF` | **厂商自定义数据** | 前 2 字节 = Company ID，后续自由定义 |

#### Flags（AD Type = 0x01）各 bit 定义

| Bit | 含义 |
|-----|------|
| bit 0 | LE 有限发现模式（Limited Discoverable Mode） |
| bit 1 | LE 普通发现模式（General Discoverable Mode） |
| bit 2 | 不支持 BR/EDR（纯 BLE 设备置 1） |
| bit 3 | Controller 同时支持 BLE 和 BR/EDR |
| bit 4 | Host 同时支持 BLE 和 BR/EDR |
| bit 5~7 | 预留 |

#### 厂商自定义数据（AD Type = 0xFF）

```
Data 结构：
┌────────────┬──────────────────────────┐
│ Company ID  │ 自定义数据                │
│ (2 bytes)   │ (N bytes, 自行定义)       │
└────────────┴──────────────────────────┘
```
前两个字节是厂商 ID（如 YiChip 的 ID 为 `0x05E`），后面内容由厂商自行定义。

#### 广播数据分析示例

原始广播数据：
```
02 01 0E 03 FF 0E 05 0D 09 59 69 63 68 69 70 20 6C 69 31 32 33
```

解析结果：

| # | Len | Type | Data | 解读 |
|---|-----|------|------|------|
| 1 | 0x02 | 0x01 (Flags) | 0x0E (`0000 1110`) | 普通发现模式 + 不支持 BR/EDR + Controller 双模 |
| 2 | 0x03 | 0xFF (厂商数据) | 0x0E 0x05 | Company ID = 0x050E（YiChip） |
| 3 | 0x0D | 0x09 (短名称) | "Yichip li123" | 设备名称缩写 |

## 七、扫描机制

### 7.1 扫描流程

```
主机 (Master/Central) 扫描过程：

┌──────────────┐
│ 设置扫描参数  │  扫描窗口 & 扫描间隔
└──────┬───────┘
       ↓
┌──────────────┐
│ 监听广播信道  │  在 CH37/38/39 上接收广播包
└──────┬───────┘
       ↓ 收到广播包
┌──────────────┐
│ 可选：发送    │  SCAN_REQ → 从机回应 SCAN_RSP（获取更多信息）
│ SCAN_REQ     │
└──────┬───────┘
       ↓
┌──────────────┐
│ 记录设备信息  │  地址、RSSI、广播数据等
│ 或发起连接    │
└──────────────┘
```

### 7.2 关键参数

| 参数 | 说明 |
|------|------|
| **扫描窗口 (Scan Window)** | 每次持续监听的时长 |
| **扫描间隔 (Scan Interval)** | 两次扫描开始之间的时间间隔 |
| **特殊情况** | 若 窗口 == 间隔，则**持续扫描**不休息 |

## 八、连接建立

### 8.1 连接请求 (CONNECT_REQ)

主机收到从机的广播后，可以发送 **CONNECT_REQ** 来发起连接请求。该请求包含以下关键参数：

| 参数 | 说明 |
|------|------|
| **连接间隔 (Conn Interval)** | 两次连接事件之间的时间（最小 7.5ms） |
| **从机延迟 (Slave Latency)** | 从机可跳过的连续连接事件数 |
| **监督超时 (Supervision Timeout)** | 最大无响应超时时间 |

### 8.2 连接后通信模型

```
连接状态下的通信（事件驱动，这是低功耗的关键）：

  时间轴 →

  ┌─ConnEvent─┐  ┌─ConnEvent─┐  ┌─ConnEvent─┐
  │ 主机唤醒   │  │ 主机唤醒   │  │ 主机唤醒   │
  │ 发送数据   │  │ 发送数据   │  │ 发送数据   │
  │ 从机应答   │  │ 从机应答   │  │ 从机应答   │
  └────┬──────┘  └────┬──────┘  └────┬──────┘
       │              │              │
       ▼              ▼              ▼
  ┌─────────────────────────────────────────┐
  │           其余时间双方都睡眠！             │  ← 低功耗核心设计
  └─────────────────────────────────────────┘
```

> ⚠️ **BLE 低功耗的本质**：不是"发射功率低"，而是**大部分时间都在睡觉**，只在约定时刻短暂唤醒交换数据。

## 九、ATT 属性协议 (Attribute Protocol)

### 9.1 核心概念

ATT 是 BLE 数据传输的桥梁，采用 **Client-Server（客户端-服务器）** 架构：

| 角色 | 通常身份 | 职责 |
|------|----------|------|
| **Server（服务器）** | 一般是从机 (Slave) | 维护一个**属性数据库(Attribute Table)** |
| **Client（客户端）** | 一般是主机 (Master) | 向 Server 发送请求来**读取/写入**属性 |

### 9.2 Attribute 属性的四要素

```
一个 Attribute 的完整结构：

┌──────────────┬────────────────┬────────────────┬───────────────┐
│ Handle       │ Type (UUID)    │ Value          │ Permissions   │
│ (句柄)       │ (类型)         │ (值)           │ (权限)        │
│ uint16_t 唯一 │ 2B 或 16B UUID │ 实际存储的数据   │ 读/写权限     │
│ "地址"       │ 定义值的含义    │                │               │
└──────────────┴────────────────┴────────────────┴───────────────┘
```

| 字段 | 说明 | 示例 |
|------|------|------|
| **Handle** | uint16 类型的唯一标识符，类似"地址"，按递增顺序存储（YiChip 要求必须**连续**） | `0x0001`, `0x0002`, `0x0003` |
| **Type (UUID)** | 全局唯一标识符，定义值的具体含义 | `0x2803`=特征声明, `0x2A37`=心率测量值 |
| **Value** | 属性的实际数据 | `"Hello"`、`25`（心率）、`85`（电量%） |
| **Permissions** | 对 Value 的访问权限 | 只读、只写、可读写、需加密 |

### 9.3 UUID 分类

| UUID 长度 | 类型 | 示例 |
|-----------|------|------|
| **16-bit** | 蓝牙 SIG 标准分配 | `0x180F`（Battery Service）、`0x2A19`（Battery Level） |
| **128-bit** | 自定义/厂商私有 | `XXXXXXXX-XXXX-XXXX-XXXX-XXXXXXXXXXXX` |

> 标准 UUID 列表查阅：[https://www.bluetooth.com/specifications/assigned-numbers](https://www.bluetooth.com/specifications/assigned-numbers)

### 9.4 MTU（最大传输单元）

| 参数 | 默认值 | 说明 |
|------|--------|------|
| **默认 MTU** | **23 字节** | ATT 层单次交互的最大数据长度 |
| **协商后 MTU** | 最高可达 **512 字节**（取决于芯片） | 连接双方交换各自支持的 MTU 后取较小值 |

> ⚠️ **未进行 MTU 交换时只能传 23 字节**，实际有效载荷更少（需扣除 ATT 操作码和 Handle 字段）。

## 十、GATT 通用属性配置文件

GATT 在 ATT 之上定义了一个 **四层树形层次框架**，将属性组织成有意义的数据结构：

```
Profile（配置文件）
  └── Service（服务）—— 相当于"文件夹"
       ├── Primary Service（主服务）
       ├── Secondary Service（次服务）
       └── Include（包含引用）
       └── Characteristic（特征）—— 相当于"文件夹里的文件"
            ├── Declaration（特征声明）
            ├── Value（特征值）← ★ 实际的应用数据在这里
            └── Descriptor(s)（描述符）—— 元数据
                 ├── CCCD (0x2902) —— 通知/指示使能
                 ├── User Description —— 用户描述字符串
                 └── Presentation Format —— 数据表示格式
```

### 10.1 四层关系详解

| 层级 | 类比 | UUID 范例 | 说明 |
|------|------|-----------|------|
| **Profile** | 一套完整的解决方案 | — | 如 HRP（心率配置）、HIDS（HID 服务） |
| **Service** | 文件夹 | `0x180D`(心率), `0x180F`(电池) | 组织相关功能，不存实际数据 |
| **Characteristic** | 文件 | `0x2A37`(心率值), `0x2A19`(电量%) | 基本数据存储单元 |
| **Descriptor** | 文件注释 | `0x2902`(CCCD) 等 | 描述特征的额外信息 |

### 10.2 Characteristic 组成

每个特征包含三个必要部分：

| 部分 | 说明 | 是否必须 |
|------|------|:--------:|
| **Characteristic Declaration** | 特征声明，包含 Handle、Properties（属性）、UUID | ✅ 必须 |
| **Characteristic Value** | 实际存储的应用数据 | ✅ 必须 |
| **Characteristic Descriptor** | 描述符（如 CCCD、用户描述等） | ⚪ 可选 |

### 10.3 Characteristic Properties（特征属性）

| Property 位 | 含义 |
|:-----------:|------|
| **Read (0x02)** | 客户端可以读取该特征值 |
| **Write (0x04)** | 客户端可以写入（无响应） |
| **Write w/o Response (0x08)** | 客户端写入无需确认 |
| **Notify (0x10)** | 服务端可主动通知客户端（需使能 CCCD） |
| **Indicate (0x20)** | 服务端主动通知 + 需客户端确认（可靠通知） |

### 10.4 CCCD（客户端特征配置描述符）

| 项目 | 说明 |
|------|------|
| **固定 UUID** | `0x2902` |
| **作用** | 使能/失能某个 Characteristic 的 Notification 和 Indication 功能 |
| **写入值** | `0x0001`= 使能 Notify, `0x0002`= 使能 Indicate, `0x0000`= 禁用 |

> 当客户端写 CCCD 使能 Notify 后，服务端的该特征值每次改变时都会**自动推送**给客户端——这就是 BLE 主动推送数据的机制！

### 10.5 GATT 属性表示例

```
Handle  | Type (UUID)     | Permissions | Value
--------|-----------------|-------------|-----------------------------
0x0001  | 0x2800 (Primary  | Read only   | 0x0018 (Heart Rate Service)
        |  Service)        |             |
0x0002  | 0x2803 (Char     | Read only   | Properties=0x12 (Notify+Read),
        |  Declaration)    |             | Handle=0x0003, UUID=0x2A37
0x0003  | 0x2A37 (Heart    | Read only   | [心率测量值，动态更新]
        |  Rate Measure)   |             |
0x0004  | 0x2902 (CCCD)    | R/W         | 0x0000 / 0x0001 / 0x0002
```

## 十一、数据收发方式

### 11.1 ATT 操作类型总览

| 操作 | 方向 | 说明 | 是否需要对方确认 |
|------|------|------|:---------------:|
| **Read Request/Response** | Client → Server | 客户端读取服务端数据 | ✅ 需要 |
| **Write Request/Response** | Client → Server | 客户端写入数据（带确认） | ✅ 需要 |
| **Write Command** | Client → Server | 写入（无响应，更快但不保证到达） | ❌ 不需要 |
| **Notification** | Server → Client | 服务端主动推送（单向） | ❌ 不需要 |
| **Indication** | Server → Client | 服务端推送（需客户端确认） | ✅ 需要 |

### 11.2 数据流完整路径示例

以手机 App 读取智能手环的心率数据为例：

```
手机 App (应用层)
    │
    ▼
GATT Client ──────────────→ GATT Server (手环)
    │  [发现服务/特征]
    ▼
ATT Client ───────────────→ ATT Server
    │  [Read Request / Write CCCD]
    ▼
L2CAP (CID=0x0004) ─────→ L2CAP
    │  [封装 ATT PDU]
    ▼
HCI ──────────────────────→ HCI
    │  [命令传输]
    ▼
LL (手机/Master) ────────→ LL (手环/Slave)
    │  [链路层打包]
    ▼
PHY (射频) ───────────────→ PHY (射频)
    │  [无线电波]
    ▼
[数据到手环] → 手环处理 → 回复原路返回
```

### 11.3 各操作详细说明

#### Read Request / Response
```
Client                        Server
  │                             │
  ├─ Read Request(handle) ────→ │  "我要读取 handle=0x0003 的值"
  │                             │
  ├─ ←── Read Response(value) ─┤  "好的，当前值是 72 bpm"
  │                             │
```

#### Write Request / Response
```
Client                        Server
  │                             │
  ├─ Write Request(handle,val)→│  "我要把 0x0005 写入 handle=0x0003"
  │                             │
  ├─ ←── Write Response ───────┤  "写入成功"
  │                             │
```

#### Notify（最常用的主动推送方式）
```
Server (手环)                  Client (手机)
  │                             │
  │  (CCCD 已被写为 0x0001)     │
  │                             │
  ├─ Notification(val) ───────→│  "心率变了！现在是 75 bpm"
  │                             │  [无需回复，单向推送]
  │                             │
```

#### Indicate（可靠的主动推送）
```
Server (手环)                  Client (手机)
  │                             │
  │  (CCCD 已被写为 0x0002)     │
  │                             │
  ├─ Indication(val) ─────────→│  "重要数据变更！"
  │                             │
  ├─ ←── Confirmation ─────────┤  "我收到了！"（确认）
  │                             │
```

## 十二、YiChip 开发实践

### 12.1 广播变量说明

```assembly
; ===== 广播配置变量 =====
mem_le_adv_enable: 01           ; 广播开关：1=开启
mem_le_adv_interval: 00 01       ; 广播间隔（单位：0.625ms）
                               ; 例：0x0100 = 256 × 0.625ms = 160ms
mem_le_lap: 11 fa 0d 43 05 13  ; 设备地址（MAC，6字节）
mem_le_name_len:               ; 设备名称长度
' Yichip li123                  ; 设备名称
mem_le_adv_data_len: 07         ; 广播数据长度
mem_le_adv_data:                ; 广报数据内容（TLV格式）
02 01 0e                       ; AD1: Flags (普通发现+不支持BR/EDR+双模Controller)
03 ff 0e 05                    ; AD2: 厂商数据 (CompanyID=0x050E=YiChip)
mem_le_scan_data_len: 0e        ; 扫描响应数据长度
mem_le_scan_data:               ; 扫描响应数据
\09 Yichip 1121s               ; AD3: 短名称 "Yichip 1121s"
mem_le_adv_channel_map: 07      ; 广播通道选择（bit0=ch37, bit1=ch38, bit2=ch39）
                               ; 0x07 = 3个通道都开
```

### 12.2 扫描变量说明

```assembly
; ===== 扫描配置变量 =====
mem_le_scan_enable: 00          ; 扫描开关：0=关闭，1=开启
mem_le_scan_type: 01            ; 扫描类型：0=被动扫描，1=主动扫描(会发SCAN_REQ)
mem_le_scan_interval: 00 02     ; 扫描间隔（单位：0.625ms）
mem_le_scan_window: 00 01       ; 扫描窗口（单位：0.625ms）
```

### 12.3 核心函数

| 函数名 | 作用 |
|--------|------|
| `le_init_adv` | 初始化广播（设置通道、数据、间隔） |
| `le_slave_dispatch` | **从机业务主函数**：处理接收/发送数据，解析 ATT 包 |
| `le_parse_att` | 解析收到的 ATT PDU，分发到对应处理函数 |
| `ui_timer_check` | 软件定时检测（基于 clkn_bt 查询） |

### 12.4 广播程序流程

```
le_init_adv()
    │
    ├── 1. 选择广播通道（channel_map → 37/38/39）
    ├── 2. 配置广播数据（adv_data → TLV 格式填充）
    ├── 3. 配置扫描响应数据（scan_rsp_data）
    ├── 4. 设置广播间隔（adv_interval → 0.625ms 单位）
    ├── 5. 使能广播（adv_enable = 1）
    │
    ▼
进入 mainloop...
    ├── 定时检查广播间隔（软件定时器）
    │   └── 到达间隔 → 触发一次广播事件（CH37→CH38→CH39）
    │
    ├── 检查是否有 SCAN_REQ 到达
    │   └── 有 → 回复 SCAN_RSP
    │
    └── 检查是否有 CONNECT_REQ 到达
        └── 有 → 进入连接状态，切换到 le_slave_dispatch
```
